# ADR-CAND-M4-001：四柱规则与权威数据绑定

**状态：** Proposed / Pending Expert Review

**类型：** ADR Candidate

**Owner Role：** Domain Architect / 命理专家

**关联需求：** FR-020、FR-021、FR-022、FR-026、FR-027、BR-008、BR-009

**上游决策：** `docs/adr/M2-DOMAIN-DECISIONS.md`、`docs/adr/ADR-CAND-M3-001-CALENDAR-PROVIDER-BINDING.md`

## Context

M4.1 需要建立四柱确定性计算边界，但 01～17 基线与现有 ADR 尚未批准以下内容：

- 年柱使用的年界与节气口径；
- 月柱的节气边界、月序与年干关联口径；
- 日柱历法数据源、锚点、历史范围与纠错策略；
- 时柱时辰边界、日干关联、子时换日与早晚子时口径；
- 真太阳时参与四柱计算的最终策略；
- 第一批专家批准黄金命例及独立交叉验证来源。

这些事项影响命盘事实，不能由工程实现根据常识或第三方库默认值静默决定。

## Problem

如何在不选择未批准命理口径的前提下，建立可复现、可版本化、逐柱可追溯的四柱计算基础，并让未来专家批准的实现可以安全接入？

## Proposed Direction（非最终决定）

1. Chart Calculation 只接受 M3 产出的单一 `EXACT` 标准化时间点。
2. 年、月、日、时每柱具有独立 `RuleId`，同时引用同一受控 `RuleSetId`、规则集版本、算法版本和数据来源。
3. 每柱规则必须引用本 ADR Candidate 及其实际适用的 M2/M3 决策；没有 ADR 引用的规则绑定不得执行。
4. 生产执行只接受 `Published` 规则与 `expert_approved` 权限清单。
5. `engineering_fixture` 仅用于契约、确定性和证据链测试，默认执行路径拒绝它；不得将其统计为专家黄金命例。
6. 未知时间、时间范围、歧义时区或未确认边界不得静默选取四柱。
7. 正式算法、数据集或外部库必须通过后续 ADR 批准，并提供来源版本、校验和、适用范围、迁移与回滚计划。

## Options Requiring Review

### Option A：受控算法实现 + 独立权威历表交叉验证

需要专家给出所有边界规则、参考命例与差异处置。可解释性强，但验证和维护成本较高。

### Option B：两个独立实现交叉验证

降低单实现缺陷风险，但两者可能共享同一错误来源；仍需专家裁决分歧。

### Option C：权威版本化结果数据集 + 受控计算实现

便于黄金与边界回归，但数据授权、覆盖范围和更新治理要求高。

当前不选择任何 Option。

## Decision Gates

正式 `expert_approved` Provider 接入前必须完成：

- [ ] 命理专家批准四柱每柱的适用规则和边界语义；
- [ ] 批准历法、时区、节气、日界与真太阳时数据来源；
- [ ] 批准子时换日和早晚子时方案，或明确不同方案的并列输出方式；
- [ ] 批准第一批黄金、边界与争议命例，记录来源和授权；
- [ ] 定义独立交叉验证实现或数据源及差异阻断阈值；
- [ ] 安全、版权与运维评审通过；
- [ ] Accepted ADR 明确迁移、回滚和 Review Trigger。

## Current Safe Behavior

- M4.1 不提供生产四柱算法；
- 只提供纯领域值、Provider Port、受控表驱动 Fixture Adapter、证据集成和确定性门禁；
- 工程候选样例明确标记 `expertApproval: pending`；
- 生产默认拒绝工程候选样例；
- 不生成五行、十神、旺衰、格局、神煞、用神、大运、流年或自然语言解读。

## Risks

- 误把工程样例当作真实命例：通过权限字段、默认拒绝和文档声明缓解。
- Provider 自报权威级别：未来组合根必须只装配受治理 Registry 中的已批准 Provider。
- 不同口径产生不同四柱：必须保留规则、参数、ADR 与源版本，禁止静默替换。
- 黄金样例不足：在专家样例完成前，不得宣称 FR-022/FR-027 已达到生产验收。

## Review Trigger

- 专家提交四柱口径或黄金命例；
- 选择生产历法、节气、时区或四柱算法 Provider；
- 子时、真太阳时或年/月边界策略发生变化；
- 引入新的算法版本、数据集或独立交叉验证源。
