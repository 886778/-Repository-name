## 1. 文档目的与规范

本文档将已确认的产品愿景落实为可执行、可测试和可追踪的软件需求。本文档规定系统行为和验收边界，不规定实现语言、框架、部署拓扑或具体数据库结构。

### 1.1 需求编号

| 前缀 | 含义 |
|---|---|
| UC | 用户故事与使用目标 |
| FR | 功能需求 |
| BR | 业务规则 |
| NFR | 非功能需求 |
| BF | 主要业务流程 |
| EF | 异常流程 |
| EC | 边界情况 |

### 1.2 优先级

| 优先级 | 定义 |
|---|---|
| Must | 所属版本不可缺少；缺失则该版本不得验收通过 |
| Should | 高价值需求；除非存在记录在案的阻塞，否则应交付 |
| Could | 有余量时交付，不阻塞所属版本发布 |
| Won’t | 明确不进入当前规划版本；保留记录以防范围漂移 |

### 1.3 版本含义

- **MVP：** 验证普通用户排盘、证据化解读和受限 AI 对话的最小正式版本。
- **V1：** 增强专业模式、多流派、完整时间轴、正式报告与多语言。
- **V2：** 研究、开发者、企业及受控插件能力。
- **不规划：** 原则上不提供的能力，以 `Won’t / 不规划` 标记。

### 1.4 验收表达

每条 FR 和 NFR 均包含可观察的验收标准。涉及具体命理判断但尚未获得专家结论时，验收对象是配置、版本、状态、追踪和禁止行为，而不是本文档自行设定命理公式。

## 2. 系统边界（System Scope）

### 2.1 系统范围内

系统负责：

1. 匿名试算、注册登录和账户数据控制；
2. 出生信息或四柱输入、校验和时间标准化；
3. 确定性排盘和内部交叉验证；
4. 版本化命理规则执行及结构化证据生成；
5. 知识检索、证据引用和来源追踪；
6. 受证据约束的 AI 解读与连续对话；
7. 在线报告、打印和后续正式报告交付；
8. 当前大运、未来三年以及后续完整时间轴；
9. 命盘、报告、对话、授权和反馈的数据管理；
10. 权限、隐私、安全、日志与审计；
11. 简体中文正式体验以及国际化、RTL 扩展能力；
12. 规则、知识、Prompt、模型和内容治理后台；
13. 商业化、开发者能力和传统术数插件的边界预留。

### 2.2 系统范围外

系统不负责：

1. 证明八字或任何传统术数具有科学预测有效性；
2. 保证未来事件、疾病、婚姻、收益或成功结果；
3. 替代医疗、法律、投资、心理等持牌专业服务；
4. 从用户反馈自动推导或修改传统命理规则；
5. 在 MVP 中处理真实支付、第三方动态插件或完整企业 API；
6. 在没有合法授权的情况下收录受版权保护的完整资料；
7. 替用户完成现实世界中的交易、签约、投资、就医或其他外部行动；
8. 在用户不知情的情况下补全不确定出生时间或变更排盘口径。

### 2.3 外部依赖

- 地点与经纬度数据服务；
- 时区、历史夏令时和历法基础数据；
- 一个或多个 AI 模型供应商；
- PostgreSQL 向量检索能力；
- 对象存储、消息通知、监控和错误追踪服务；
- V1 可能接入的 PDF 生成与安全分享能力；
- V2 可能开放的 API、Webhook 和 SDK。

任何外部依赖失败均不得改变已经计算并保存的确定性结果；系统应按异常流程降级或明确失败。

## 3. 角色与权限概览

| 角色 | 说明 | 核心权限 |
|---|---|---|
| AnonymousUser | 未注册且已完成必要告知的成年用户 | 创建临时命盘、查看临时报告、有限 AI 对话、打印、反馈 |
| RegisteredUser | 已注册普通用户 | 管理本人命盘、报告、对话、导出、删除和授权 |
| ProfessionalUser | 获得专业模式权限的用户 | 展开计算参数、规则依据、版本及 V1 多流派能力 |
| ContentEditor | 内容编辑 | 创建知识、术语、Prompt 草稿，不可自行发布生产版本 |
| DomainExpert | 命理专家 | 审核规则和领域知识，记录同意、反对或分歧意见 |
| Reviewer | 发布审核者 | 审核规则、知识、Prompt 和高风险内容版本 |
| SupportAgent | 客服或支持人员 | 在用户授权及审计下查看最小必要信息，处理反馈与删除请求 |
| SystemAdministrator | 系统管理员 | 配置系统和权限；默认不可浏览明文敏感业务内容 |
| Auditor | 审计人员 | 只读查看审计记录和版本链，不可修改业务数据 |
| DeveloperClient | V2 API 客户 | 在密钥、租户、额度和版本限制内调用开放能力 |

权限细节以最小权限、职责分离和敏感操作复核为原则。

## 5. 主要业务流程（Business Flows）

### BF-001 匿名出生信息排盘与首次报告

1. 用户进入排盘页并完成 18 岁确认、必要服务告知和隐私告知。
2. 用户选择公历或农历，填写出生日期、时间、时间精度、地点及算法所需传统参数。
3. 系统校验字段、解析地点、推断时区并识别潜在边界。
4. 系统展示标准化结果、真太阳时状态、换日规则和所有关键参数。
5. 用户确认参数并提交。
6. 排盘引擎生成带版本的确定性事实。
7. 内部校验对关键结果执行一致性、范围和必要的交叉验证。
8. 规则引擎只运行已发布且适用的规则集，生成结构化结论与证据。
9. AI 编排器组装去标识化证据，生成受结构约束的解释。
10. 系统执行事实一致性、引用、冲突和风险检查。
11. 检查通过后显示首屏摘要和可展开依据；失败则执行 EF-005 或 EF-006。
12. 匿名用户可打印、反馈、继续有限对话或注册保存。

**成功条件：** 命盘、证据和报告版本链完整；用户看到结果；没有把不确定输入表示为确定事实。  
**失败条件：** 关键输入无法解析、算法校验失败或内容安全检查未通过且无法降级。

### BF-002 直接四柱创建命盘

1. 用户选择“直接输入四柱”。
2. 系统校验四个干支组合的格式和合法性。
3. 用户确认该模式不能自动恢复未提供的精确出生时间、地点和历法参数。
4. 系统保存来源类型为 `manual_pillars` 的命盘输入。
5. 系统计算只依赖四柱即可确定的派生事实。
6. 对依赖缺失出生信息的功能显示“不适用”或要求补充信息。
7. 后续规则、AI 和报告流程与 BF-001 共用证据与安全检查。

### BF-003 AI 命理对话

1. 用户从某一有效命盘或报告进入对话。
2. 系统识别问题是否属于当前命盘、未来三年和支持主题。
3. 系统选择与问题相关的排盘事实、规则证据和已审核知识。
4. 系统去除直接身份信息，构造本轮模型上下文。
5. 模型返回结构化回答草稿及证据引用。
6. 系统验证引用、事实、时间范围、冲突表达和高风险边界。
7. 通过则展示回答；不通过则有限重试、使用安全模板降级或拒绝。
8. 系统记录对话版本和审核结果，不记录未获准的额外身份信息。

### BF-004 报告保存与历史复现

1. 报告生成成功后冻结报告内容和生成快照。
2. 快照至少关联输入、标准化参数、算法、规则集、知识索引版本、Prompt 和模型标识。
3. 后续规则或模型更新创建新版本，不覆盖原报告。
4. 用户查看旧报告时看到原报告及其版本信息。
5. 用户主动要求重新分析时创建新分析和新报告，并可与旧版区分。

### BF-005 规则与知识发布

1. 有权限的编辑者创建草稿并填写来源、适用条件、流派、版本说明和影响范围。
2. 自动检查必填元数据、引用有效性和兼容性。
3. 命理专家审核领域内容；版权或高风险内容按需进入专项审核。
4. 独立审核者批准、拒绝或要求修改。
5. 批准后生成不可变发布版本，并记录发布人、时间和变更说明。
6. 新版本仅影响其生效时间之后的新分析，旧报告保持原快照。
7. 出现问题时停用或回滚到既有发布版本，操作写入审计日志。

### BF-006 用户数据删除和授权撤回

1. 用户发起删除账户、删除单个命盘或撤回命例优化授权。
2. 系统完成身份核验并展示影响范围。
3. 系统创建可追踪的数据权利请求。
4. 在线业务数据按策略删除或去标识化；受法定义务保留的数据进入受限状态。
5. 对象存储、检索索引、分析数据和异步任务同步处理。
6. 备份数据按既定保留周期自然到期，不再恢复为活动数据。
7. 系统向用户提供完成状态；失败项进入人工处置并审计。

### BF-007 V1 多流派比较

1. 用户选择已发布的两个或多个流派规则集。
2. 各规则集基于同一确定性命盘快照独立运行。
3. 系统按结论主题对齐结果，标记一致、差异、冲突和不适用。
4. AI 只比较已有证据，不裁定未经确认的唯一正确流派。
5. 报告保存所有参与规则集及版本。

## 6. 功能需求：用户系统与账户

### FR-001 匿名试算

- **优先级：** Must
- **版本：** MVP
- **需求：** 用户无需注册即可创建临时命盘、查看在线报告、进行受限 AI 对话和打印。
- **验收标准：**
  1. 未登录用户完成必要告知后可走通 BF-001。
  2. 系统不要求姓名、手机号或邮箱作为试算前提。
  3. 匿名记录带明确过期状态，不进入永久用户资产列表。
  4. 匿名数据到期策略可配置并可在测试环境验证清理结果。

### FR-002 年龄确认

- **优先级：** Must
- **版本：** MVP
- **需求：** 独立用户在首次使用前确认年满 18 岁。
- **验收标准：** 未确认或明确未满 18 岁时，系统不允许创建命盘；处理方式、文案和是否需要进一步年龄保障机制标记为法律待确认。

### FR-003 注册与登录

- **优先级：** Must
- **版本：** MVP
- **需求：** 用户可注册、登录、退出和安全恢复账户，以保存本人资产。
- **验收标准：**
  1. 注册不是匿名试算前置条件。
  2. 登录后只能访问所属账户或明确获授权的数据。
  3. 恢复账户需通过已验证渠道，不泄露账户是否存在的敏感细节。
  4. 所有会话可由用户主动终止，安全事件时管理员可撤销会话。

### FR-004 匿名资产转存

- **优先级：** Should
- **版本：** MVP
- **需求：** 匿名用户注册后可将当前临时命盘和报告转入账户。
- **验收标准：** 只有持有当前匿名会话凭证的用户可转存；重复操作不创建重复资产；过期资产不可被其他账户认领。

### FR-005 账户设置

- **优先级：** Must

### FR-031 子平基础规则集

- **优先级：** Must
- **版本：** MVP
- **需求：** 支持第一套经专家批准的子平八字基础规则集。
- **验收标准：** 只有专家审核通过的规则可发布；旺衰、格局、调候、用神等未确认规则保持未发布或实验状态；系统能生成规则版本清单。

### FR-032 规则状态与适用性

- **优先级：** Must
- **版本：** MVP
- **需求：** 规则结果支持“满足、不满足、不适用、信息不足、存在冲突、执行错误”状态。
- **验收标准：** 所有状态可被报告和 AI 正确消费；信息不足不显示为否定结论；执行错误进入监控且不转化为命理判断。

### FR-033 结构化证据

- **优先级：** Must
- **版本：** MVP
- **需求：** 每个重要规则结论生成结构化证据，包括事实引用、规则引用、适用条件、流派、版本和证据状态。

### FR-034 证据等级

- **优先级：** Must
- **版本：** MVP
- **需求：** 使用“充分、一般、有限、存在冲突”表达规则支持状态。
- **验收标准：** 等级计算依据可追踪且由配置版本控制；界面明确说明其不是事件概率；具体分级规则待命理专家与产品确认。

### FR-035 规则冲突

- **优先级：** Must
- **版本：** MVP
- **需求：** 系统能够保留互相冲突的规则结论，而不是静默覆盖。
- **验收标准：** 冲突双方均可追踪；报告显示冲突状态；AI 不得自行裁定唯一正确观点；多流派优先级为专家待确认。

### FR-036 规则版本发布

- **优先级：** Must
- **版本：** MVP
- **需求：** 规则支持草稿、审核、批准、发布、停用和回滚的受控生命周期。
- **验收标准：** 创建者不可单独完成需职责分离的发布；每次状态变更有理由和审计记录；回滚只影响新分析；旧报告保持原版本。

### FR-037 首批神煞

- **版本：** MVP
- **需求：** 系统仅运行进入专家批准清单的少量神煞规则。
- **验收标准：** 未批准神煞不出现在正式报告；每项标记来源、适用规则和允许影响范围；首批清单与权重待专家确认。

### FR-038 多流派并行分析

- **优先级：** Must
- **版本：** V1
- **需求：** 多个已发布流派规则集可基于同一命盘快照独立运行并比较。
- **验收标准：** 不共享会改变结论的隐式状态；展示一致、差异、冲突和原因；AI 比较不能掩盖原始观点；具体冲突优先级待专家确认。

- **版本：** MVP

### FR-045 匿名命例入库

## 11. 功能需求：AI 解读

### FR-050 分析计划

- **优先级：** Must
- **版本：** MVP
- **需求：** AI 生成前，系统根据报告类型和可用证据建立明确的分析主题与证据范围。
- **验收标准：** 计划不包含超出已支持主题的项目；缺少证据的主题被省略或标记信息不足；计划版本被记录。

### FR-051 去标识化上下文

- **优先级：** Must
- **版本：** MVP
- **需求：** 第三方 AI 只接收完成解读所需的去标识化命盘事实、规则证据和知识片段。
- **验收标准：** 请求中不包含姓名、联系方式、账户 ID、详细地址或无必要的原始地点文本；自动测试检查敏感字段；例外需明确授权和安全评审，MVP 默认不允许。

### FR-052 结构化 AI 输出

- **优先级：** Must
- **版本：** MVP
- **需求：** AI 输出必须符合预定义结构，区分结论、依据、综合解释、建议、不确定性和风险提示。
- **验收标准：** 结构不合法不得直接展示；系统执行有限重试；仍失败时使用安全降级结果或报告生成失败。

### FR-053 事实一致性检查

- **优先级：** Must
- **版本：** MVP
- **需求：** 展示前验证 AI 提及的四柱、十神、时间节点及其他事实与证据包一致。
- **验收标准：** 注入错误事实的测试输出被拦截；AI 不得创建新的事实 ID；无法自动验证的事实性陈述不得作为确定结果展示。

### FR-054 引用检查

- **优先级：** Must
- **版本：** MVP
- **需求：** 重要 AI 结论必须引用存在且相关的证据。
- **验收标准：** 不存在、无权限、版本不匹配或不支持结论的引用被拒绝；正式报告可统计引用有效率；一般性低风险建议可不声称命理依据但必须标记为现实建议。

### FR-055 冲突与不确定性表达

- **优先级：** Must
- **版本：** MVP
- **需求：** AI 根据证据状态使用相应措辞，不把冲突或有限证据表达为确定结论。
- **验收标准：** 针对充分、一般、有限、冲突和无证据分别建立评测样例；冲突样例必须提及分歧；无证据样例必须拒绝补造。

### FR-056 风险与合规检查

- **优先级：** Must
- **版本：** MVP
- **需求：** AI 内容展示前经过绝对化预测、恐吓、医疗、法律、投资、赌博、未成年人及其他高风险检查。
- **验收标准：** 已知风险测试集达到发布门槛；命中后可改写、拒绝或引导专业帮助；原始输出和处置结果按隐私策略审计；最终边界待法律确认。

### FR-057 模型路由和降级

- **优先级：** Must
- **版本：** MVP
- **需求：** 系统可按任务、语言、成本、安全和可用性选择模型，并在超时或失败时降级。
- **验收标准：** 模型供应商不可用时确定性排盘仍可展示；降级路径不跳过安全检查；记录供应商、模型标识、重试次数、耗时和成本。

### FR-058 Prompt 版本管理

- **优先级：** Must
- **版本：** MVP
- **需求：** 所有生产 Prompt 通过受控版本发布，报告记录所用版本。
- **验收标准：** 草稿 Prompt 不进入生产；发布和回滚可审计；旧报告不因 Prompt 更新而改变。

### FR-059 多模型复核

- **优先级：** Could
- **版本：** V1
- **需求：** 对高价值或高风险分析可选择第二模型复核，不把模型投票当作事实证明。
- **验收标准：** 复核范围和触发条件可配置；成本单独统计；模型分歧被记录并进入保守输出或人工审查。

- **版本：** MVP
- **需求：** AI 对话限定在当前命盘、未来三年和已支持主题。
- **验收标准：** 超出命盘、时间范围或主题的问题被说明边界并安全拒绝或引导；系统不会为回答越界问题临时生成未支持规则。

### FR-061 多轮上下文

- **优先级：** Must
- **版本：** MVP
- **需求：** 在同一命盘会话中保留必要的多轮语义，同时每轮重新绑定当前有效证据范围。
- **验收标准：** 切换命盘后不会引用前一命盘事实；超过上下文上限时安全摘要；摘要不得改变关键事实或证据 ID。

### FR-062 对话证据引用

- **优先级：** Must
- **版本：** MVP
- **需求：** 对话中的命理结论必须提供可展开证据。
- **验收标准：** 引用检查规则与正式报告一致；用户可从引用进入通俗依据，专业用户可继续查看规则和版本。

### FR-063 高风险问答处置

- **优先级：** Must
- **版本：** MVP
- **需求：** 对医疗诊断、具体投资指令、法律结论、赌博、伤害、自杀、死亡预测等请求进行专项处置。
- **验收标准：** 风险测试问题不会得到替代专业意见或必然预测；必要时提供紧急或专业支持提示；最终类别、措辞和地区资源待法律与安全专家确认。

### FR-064 对话限额

- **优先级：** Must
- **版本：** MVP
- **需求：** 匿名和注册用户的轮数、频率、上下文和成本设有可配置限额。
- **验收标准：** 达限时明确提示且不丢失已生成报告；限额变更不回溯改变已消费记录；管理员可查看聚合成本，不默认查看私人对话正文。

### FR-065 对话反馈

- **优先级：** Should
- **版本：** MVP
- **需求：** 用户可对单条回答标记有帮助、无帮助、解释不清、依据疑问或风险问题。
- **验收标准：** 反馈关联回答、模型与版本；不自动修改规则；用于优化需要符合授权和数据用途限制。

### FR-076 多语言报告

- **版本：** V1

### FR-090 数据查看与导出

### FR-094 数据保留策略

- **验收标准：** 到期任务可测试；保留变更有审计；法律冻结优先但权限受限；具体期限在上线前确认。

### FR-095 第三方处理透明度

## 16. 功能需求：权限、日志与审计

### FR-100 角色与最小权限

- **优先级：** Must

- **验收标准：** 对每类角色完成允许与拒绝测试；跨账户直接对象访问被拒绝；权限变更即时或在定义时间内生效并审计。

### FR-101 职责分离

- **优先级：** Must
- **版本：** MVP
- **需求：** 规则、知识、Prompt 和高风险配置的生产发布支持创建与审核分离。
- **验收标准：** 同一用户不能在要求双人审核的对象上完成全部步骤；紧急变更需特殊权限、理由和事后复核。

### FR-102 业务审计日志

- **优先级：** Must
- **版本：** MVP
- **需求：** 记录登录安全事件、权限变化、敏感访问、版本发布、回滚、删除、导出、同意变化和管理配置变更。
- **验收标准：** 每条日志含主体、动作、对象、结果、时间和关联标识；日志不可被普通管理员修改；可按事件调查检索。

### FR-103 系统运行日志

- **优先级：** Must
- **版本：** MVP
- **需求：** 记录请求、任务、错误、外部依赖、模型调用和性能信息，同时避免写入不必要敏感正文。

### FR-104 报告复现审计

- **优先级：** Must
- **版本：** MVP
- **需求：** 审计角色可查看某报告的完整版本链和生成状态。
- **验收标准：** 能定位输入、参数、算法、规则、知识、Prompt、模型、安全检查和生成时间；不能通过审计界面修改原数据。

### FR-105 安全事件记录

- **优先级：** Must
- **版本：** MVP
- **需求：** 对异常登录、越权尝试、批量访问、数据导出和管理操作生成安全事件。
- **验收标准：** 预设测试事件可触发记录或告警；事件含严重度和处置状态；告警失败不阻断审计记录写入。

## 19. 功能需求：后台管理

### FR-130 用户与访问管理

- **优先级：** Must
- **版本：** MVP
- **需求：** 授权管理员可查找账户状态、冻结异常会话、处理数据请求和查看最小必要信息。
- **验收标准：** 敏感字段默认遮蔽；每次查看和操作有审计；管理员不能模拟用户而不留下明确记录。

### FR-131 命盘与报告支持视图

- **优先级：** Must
- **版本：** MVP
- **需求：** 支持人员在工单或用户授权范围内查看命盘处理状态、版本和错误，不默认查看全部正文。
- **验收标准：** 访问范围随授权到期；跨工单访问被拒绝；排障数据可与关联 ID 对齐。

### FR-132 规则管理

- **优先级：** Must
- **版本：** MVP
- **需求：** 后台支持规则草稿、元数据、专家意见、审核、发布、停用、回滚和影响范围查看。

### FR-133 知识与版权管理

- **优先级：** Must
- **版本：** MVP

### FR-134 Prompt 与模型管理

- **版本：** MVP
- **需求：** 管理 Prompt 版本、模型供应商、模型标识、任务路由、成本上限、超时、重试和降级。
- **验收标准：** 配置变更需要授权并审计；无效配置不能发布；密钥不在界面或日志明文显示；测试与生产配置隔离。

### FR-135 反馈与投诉管理

- **优先级：** Must
- **版本：** MVP
- **需求：** 后台对反馈按数据错误、规则疑问、解释不清、风险内容和隐私请求分类处理。
- **验收标准：** 每项有状态、责任人和处理记录；高风险类别可升级；反馈不能直接触发生产规则修改。

### FR-136 版本影响分析

- **优先级：** Should
- **版本：** MVP
- **需求：** 发布前显示规则、知识、Prompt 或模型变更可能影响的主题和新分析范围。
- **验收标准：** 至少列出依赖对象和测试结果；不会修改旧报告；复杂历史模拟作为 V1/V2 增强项。

### FR-137 套餐与额度预留

- **优先级：** Could
- **版本：** MVP
- **需求：** 允许配置非支付性质的免费额度和功能开关，为后续商业化预留边界。
- **验收标准：** MVP 不产生真实订单或资金流水；额度扣减可审计且幂等；商业化接口不影响匿名基础告知和数据权利。

## 21. 非功能需求（Non-functional Requirements）

### NFR-001 首次任务效率

- **优先级：** Must
- **版本：** MVP
- **要求：** 普通用户主路径以三分钟内完成首次排盘为目标。
- **验收标准：** 在目标用户可用性测试中记录从开始输入到首屏摘要可见的时间；测试方法、样本量和通过比例在原型阶段确认，未达到门槛不得宣称达成目标。

### NFR-002 页面响应

- **优先级：** Must
- **版本：** MVP
- **要求：** 除报告生成外的常规交互应保持可感知流畅。
- **验收标准：** 在约定基准设备和网络下，核心页面性能指标及 API 百分位目标在架构评审前量化；超时必须显示进度或可恢复错误，不允许无限等待。

### NFR-003 排盘性能

- **优先级：** Must
- **版本：** MVP
- **要求：** 确定性排盘不依赖 AI，可独立完成并返回。
- **验收标准：** AI 供应商完全不可用时仍能生成排盘事实；具体 P95 时延与并发目标在容量估算后写入测试基线。

### NFR-004 AI 时延与超时

- **优先级：** Must
- **版本：** MVP
- **要求：** AI 报告和对话具有明确超时、有限重试和降级路径。
- **验收标准：** 模拟超时、限流、结构错误和供应商故障；每种场景在预定时间内成功、降级或明确失败；不发生无限重试。

### NFR-005 可用性

- **优先级：** Must
- **版本：** MVP
- **要求：** 定义正式服务可用性目标和计划维护规则。
- **验收标准：** MVP 上线前确定月度可用性 SLO、排除项和测量来源；排盘、AI 和后台分别监控，避免 AI 故障被误算为全部服务故障。

### NFR-006 数据一致性

- **优先级：** Must
- **版本：** MVP
- **要求：** 命盘、分析、报告和版本快照之间保持引用一致性。
- **验收标准：** 事务失败、重复提交和任务重试测试不会产生半完成正式报告或重复扣减；孤立记录可检测和修复。

### NFR-007 幂等性

- **优先级：** Must
- **版本：** MVP
- **要求：** 创建计算任务、报告任务、额度扣减和删除请求支持幂等处理。
- **验收标准：** 相同幂等标识重复提交不会产生重复正式对象或重复扣减；冲突请求返回可诊断结果。

### NFR-008 可复现性

- **优先级：** Must
- **版本：** MVP
- **要求：** 正式报告的结构化事实和规则结论可按原版本复现。
- **验收标准：** 从抽样报告恢复全部必要版本并重放；结果与冻结快照一致；无法复现视为阻断级质量问题。

### NFR-009 安全传输与存储

- **优先级：** Must
- **版本：** MVP
- **要求：** 敏感数据传输和存储使用行业通行的加密保护，密钥独立管理。
- **验收标准：** 生产通信不允许明文敏感传输；备份和对象存储受加密与访问控制保护；密钥不进入源码、日志或报告。

### NFR-010 身份与会话安全

- **优先级：** Must
- **版本：** MVP
- **要求：** 防止会话固定、凭证暴力尝试和未授权会话延续。
- **验收标准：** 登录限速、会话轮换、退出失效、密码或身份恢复安全测试通过；高权限账户多因素认证策略在上线前确认。

### NFR-011 授权隔离

- **优先级：** Must
- **版本：** MVP
- **要求：** 所有用户资源访问在服务端校验归属或授权。
- **验收标准：** 跨用户、越权角色、猜测对象 ID 和失效分享凭证测试全部被拒绝并记录安全事件。

### NFR-012 隐私保护

- **优先级：** Must
- **版本：** MVP
- **要求：** 遵循目的限制、数据最小化、默认保护和生命周期管理。
- **验收标准：** 数据清单可映射到处理目的；非必要字段不收集；第三方请求通过去标识化检查；删除和保留测试通过。

### NFR-013 审计完整性

- **优先级：** Must
- **版本：** MVP
- **要求：** 关键审计记录防篡改、按权限只读并可关联调查。
- **验收标准：** 普通管理员无法修改或删除；审计存储失败时高风险管理操作被阻断或触发明确告警；时间来源一致。

### NFR-014 可观察性

- **优先级：** Must
- **版本：** MVP
- **要求：** 排盘、规则、检索、AI、报告和后台任务均具备日志、指标和错误追踪。
- **验收标准：** 可通过关联 ID 定位一次失败流程；定义成功率、时延、错误率、队列积压、模型成本和安全检查指标；告警责任人明确。

### NFR-015 备份与恢复

- **优先级：** Must
- **版本：** MVP
- **要求：** 数据库、对象和关键配置有自动备份及恢复验证。
- **验收标准：** MVP 上线前确定 RPO/RTO；至少完成一次隔离环境恢复演练；恢复不会使已删除数据重新进入活动服务。

### NFR-016 可维护性

- **优先级：** Must
- **版本：** MVP
- **要求：** 领域模块边界清晰，排盘、规则、AI、报告、隐私和审计可独立测试与演进。
- **验收标准：** 架构文档给出依赖约束；模块接口有契约测试；AI 供应商替换不改变排盘事实模型。

### NFR-017 兼容性与移动端

- **优先级：** Must
- **版本：** MVP
- **要求：** 核心流程优先支持主流移动设备和现代浏览器。
- **验收标准：** 上线前确定支持矩阵；最小支持视口下无需横向滚动完成表单和阅读摘要；专业宽表有可用的移动呈现方式。

### NFR-018 可访问性

- **优先级：** Must
- **版本：** MVP
- **要求：** 核心用户旅程以 WCAG 2.2 AA 为目标。
- **验收标准：** 自动扫描加人工键盘和屏幕阅读器测试；阻断级问题在发布前清零；无法立即满足的项记录例外、风险和修复期限。

### NFR-019 国际化

- **优先级：** Must
- **版本：** MVP
- **要求：** 文本、布局方向、日期数字和术语资源可本地化。
- **验收标准：** 伪本地化测试无关键截断；RTL 测试覆盖核心组件；业务存储不依赖显示语言。

### NFR-020 AI 成本控制

- **优先级：** Must
- **版本：** MVP
- **要求：** 按报告、对话轮次、模型和用户类型测量并限制 AI 成本。
- **验收标准：** 每次调用记录可归集用量和估算成本；超出预算策略时降级或拒绝；成本限制不绕过安全检查；目标单价待模型选型确认。

### NFR-021 可扩展容量

- **优先级：** Should
- **版本：** MVP
- **要求：** 模块化单体支持通过无状态应用实例和独立后台任务扩展，不预设微服务。
- **验收标准：** 容量测试目标在架构阶段根据预估用户量确定；扩容不破坏任务幂等和版本一致性。

### NFR-022 版权与内容治理

- **优先级：** Must
- **版本：** MVP
- **要求：** 知识和报告内容遵循来源、授权、引用范围与撤下流程。
- **验收标准：** 未授权条目不能进入生产检索；可按来源定位受影响内容；收到有效撤下要求后可阻止新生成并保留必要审计。

### NFR-023 灾难与供应商降级

- **优先级：** Must
- **版本：** MVP
- **要求：** 单一 AI、地点或监控供应商故障不应破坏已保存命盘和报告。
- **验收标准：** 故障演练证明已保存内容可读；新请求明确降级或失败；不会切换到未经批准的供应商。

### NFR-024 测试与发布门禁

- **优先级：** Must
- **版本：** MVP
- **要求：** 发布需通过排盘黄金样例、规则回归、权限、隐私、AI 事实引用和风险测试。
- **验收标准：** 门禁项、阈值、例外审批和证据可审计；任何关键门禁未通过时生产发布被阻止。具体阈值在专家和架构评审后确认。

## 22. 异常流程（Exception Flows）

### EF-001 地点无法解析

- 系统展示候选地点或要求用户提供更高层级行政区信息。
- 未经用户确认不得自动选择同名地点。
- 仍无法解析时，可允许直接四柱模式或终止出生信息排盘。

### EF-002 本地时间无效或重复

- 对历史夏令时导致的不存在时间或重复时间，系统展示冲突及候选偏移。
- 用户确认前不得生成正式命盘。
- 选择和数据源版本写入快照。

### EF-003 出生时间跨越关键边界

- 系统识别时间范围是否跨时柱、日期、子时换日或节气边界。
- 若可能产生多个命盘，显示显著警告；不得用单一确定报告隐藏差异。
- MVP 是生成候选盘还是阻断部分分析待产品与专家确认。

### EF-004 排盘交叉验证不一致

- 记录字段级差异和参与版本。
- 已有批准差异说明时可继续并展示说明。
- 未解释的关键差异阻断正式报告，转入专家或技术复核。

### EF-005 规则执行失败

- 已成功的确定性排盘仍可展示。
- 失败规则不生成结论，状态标记为执行错误。
- 报告不得让 AI 补足缺失结论；系统记录错误并告警。

### EF-006 AI 生成或校验失败

- 对超时、无效结构、事实错误或引用错误执行有限重试。
- 重试失败后展示确定性排盘和已验证规则摘要，或提示报告稍后重试。
- 不展示未通过检查的原始 AI 内容。

### EF-007 高风险问题

- 系统识别风险类型并采用安全回答、拒绝或专业支持引导。
- 不使用命理内容给出诊断、具体投资指令或必然灾祸判断。
- 处置记录遵守最小化和保留策略。

### EF-008 用户会话过期

- 未提交输入在本地或服务端是否暂存按隐私策略处理。
- 重新认证后只恢复用户本人可访问的安全状态。
- 匿名会话过期后不得被新会话认领。

### EF-009 删除任务部分失败

- 请求状态显示“处理中”或“需人工处理”，不虚假报告完成。
- 系统重试可恢复步骤并保持幂等。
- 无法处理的存储进入人工工单和审计。

### EF-010 知识授权失效

- 失效条目立即停止进入新检索。
- 已冻结旧报告保留引用审计，但对外展示方式按法律意见处理。
- 系统列出受影响的新生成模板和待处理内容。

### EF-011 模型供应商不可用

- 排盘和已保存报告继续可用。
- 仅切换到已批准供应商；无安全替代时停止 AI 生成。
- 用户得到明确状态，不重复扣减额度。

### EF-012 权限或越权访问

- 返回不泄露资源存在性的安全错误。
- 记录安全事件并按严重度告警。
- 不在客户端错误中暴露内部标识、权限规则或敏感数据。

## 23. 边界情况（Edge Cases）

| 编号 | 边界情况 | 预期系统行为 |
|---|---|---|
| EC-001 | 出生时刻恰在节气边界前后 | 使用版本化精确边界；显示边界提示；黄金样例验证。 |
| EC-002 | 23:00 至次日 01:00 附近出生 | 按用户确认的换日方案计算；展示差异风险；不宣称唯一口径。 |
| EC-003 | 地点曾变更行政区名称 | 使用时间相关或可追踪地点数据；保存标准化标识和来源。 |
| EC-004 | 历史时区与当前时区不同 | 使用出生时刻对应的历史规则，不直接套用当前偏移。 |
| EC-005 | 夏令时开始导致本地时间不存在 | 阻止静默计算，要求选择或补充证据。 |
| EC-006 | 夏令时结束导致本地时间重复 | 展示两个候选偏移，用户确认后保存选择。 |
| EC-007 | 出生时间只有“上午/晚上” | 保存范围；若跨时柱则不输出单一确定时柱结论。 |
| EC-008 | 直接四柱输入但无出生地点 | 只运行不依赖地点和时区的能力，其余标记不适用。 |
| EC-009 | 用户切换真太阳时后跨时柱 | 明确显示命盘发生变化并要求重新确认。 |
| EC-010 | 农历闰月 | 要求明确闰月状态；转换结果供用户确认；样例回归。 |
| EC-011 | 极端经度或海外出生地 | 使用对应时区和经纬度；超出已验证范围时提示或转人工验证。 |
| EC-012 | 规则同时得出相反观点 | 保留双方证据，状态为存在冲突，AI 不作唯一裁决。 |
| EC-013 | 证据支持结论但知识来源授权过期 | 新报告不得引用失效知识；规则事实与知识解释分开处置。 |
| EC-014 | 模型返回不存在的证据 ID | 校验失败，重试或降级，绝不直接展示。 |
| EC-015 | 用户问第四年趋势 | MVP 说明三年范围，不临时绕过范围生成。 |
| EC-016 | 用户要求预测死亡、疾病或具体收益 | 进入高风险流程，不给必然结论或替代专业意见。 |
| EC-017 | 用户保存他人命盘 | 要求确认具有合法授权；在界面和数据模型中标记资料关系。 |
| EC-018 | 同一请求被连续点击 | 幂等处理，不生成重复报告或重复消耗额度。 |
| EC-019 | 报告生成期间规则发布新版本 | 当前任务继续使用启动时锁定版本；新任务使用生效版本。 |
| EC-020 | 用户删除命盘时存在报告和对话 | 展示关联影响，并按选择和保留规则级联处理或阻止。 |
| EC-021 | 分享链接被撤销后缓存仍存在 | V1 撤销后服务端访问立即失败；缓存不得绕过授权。 |
| EC-022 | RTL 页面包含四柱表和时间轴 | 保持术语顺序语义正确，方向由组件级规则控制并人工验收。 |
| EC-023 | 用户撤回研究授权时数据已进入研究集 | 停止后续使用并执行已确认撤回策略；具体历史处理待法律确认。 |
| EC-024 | 算法版本已停用但旧报告需查看 | 旧报告快照可读；停用版本不得用于新计算。 |
| EC-025 | 多流派对同一概念命名不同 | 使用稳定概念标识，分别展示流派术语，不强制错误合并。 |

## 24. MVP 总体验收条件

MVP 只有同时满足以下条件才可进入正式上线评审：

1. FR 中标记为 `Must / MVP` 的需求全部通过，或具有经产品、安全和相关专家共同批准的书面例外。
2. 普通用户可以匿名完成输入、参数确认、排盘、报告、证据展开、受限对话和打印。
3. 确定性排盘在 AI 完全不可用时仍可工作。
4. 专家批准的黄金命例和边界样例达到待确认的发布门槛。
5. 旺衰、格局、调候、用神、起运和神煞等未确认规则没有进入正式生产规则集。
6. AI 事实一致性、引用有效性和高风险测试达到发布门槛。
7. 报告能够定位完整输入与版本快照，旧报告不被升级覆盖。
8. 匿名数据过期、用户导出、删除和授权撤回流程通过端到端测试。
9. 跨账户、越权、敏感后台访问和审计完整性测试通过。
10. 简体中文移动端主流程、基础可访问性和打印完成验收。
11. 监控、告警、备份恢复和供应商降级完成演练。
12. 中国大陆目标市场的隐私、内容、版权和法律专项评审完成，高风险遗留项有明确处置。
13. 性能、可用性、成本和保留期中本文档尚未量化的目标，在架构和上线评审前获得正式数值。

## 25. 需求追踪与变更控制

1. 每个设计、实现任务和测试用例应引用至少一个 FR、BR 或 NFR 编号。
2. 每条 Must 需求必须具有正向、异常和权限测试；涉及敏感数据者还需隐私测试。
3. 涉及命理结论的测试必须标注专家样例版本，不得由研发自行定义期望结论。
4. 涉及 AI 的测试应区分确定性校验、统计评测和人工抽样，不把单次模型输出当作稳定契约。
5. 需求变更需记录原因、影响版本、数据迁移影响、旧报告复现影响和安全影响。
6. 从 Must 降级、从后续版本提前或扩大数据用途均需重新评审。
7. 本 SRS 通过后，架构文档应逐条说明高风险 NFR 和跨模块 FR 的实现责任边界，但仍不进入代码实现。

## 25A. MVP 初始性能与容量基线

以下数字是架构设计和首次容量测试使用的初始基线，不是对外服务承诺。若真实压测或模型选型表明需要调整，必须通过需求变更记录更新，不得由实现人员静默放宽。

### NFR-025 确定性排盘时延

- **优先级：** Must
- **版本：** MVP
- **要求：** 在基准负载、缓存未命中且外部地点解析已完成的条件下，确定性排盘 P95 不超过 2 秒，P99 不超过 5 秒。
- **验收标准：** 使用正式黄金命例混合数据执行不少于 1,000 次测试；报告 P50、P95、P99、错误率和测试环境；AI 时延不得计入排盘时延。

### NFR-026 常规服务响应

- **优先级：** Must
- **版本：** MVP
- **要求：** 不包含 AI、PDF、地点外部查询和异步报告生成的已认证常规 API，在 MVP 基准负载下 P95 不超过 500 毫秒，P99 不超过 1.5 秒。
- **验收标准：** 按资源类型分别统计，不能用快速健康检查稀释慢请求；错误请求与成功请求分开报告。

### NFR-027 AI 首次有效响应

- **优先级：** Must
- **版本：** MVP
- **要求：** AI 对话从服务端接受请求到首个通过展示条件的有效响应，P95 目标不超过 15 秒；1 秒内应返回已受理或进度状态。完整报告生成 P95 初始目标不超过 60 秒。
- **验收标准：** 统计必须包含模型调用、结构校验、事实与引用检查；超时按 EF-006 降级；分别报告成功、降级和失败样本。该目标须在模型选型后复核。

### NFR-028 MVP 峰值容量

- **优先级：** Must
- **版本：** MVP
- **要求：** 初始架构基线支持 100 个并发交互请求、20 个并发 AI 任务和持续 10 请求/秒的核心混合负载，且不破坏幂等、权限或版本一致性。
- **验收标准：** 压力测试持续至少 30 分钟；错误率不超过 1%，Must 级时延目标的违反情况需单独报告；超过设计容量时系统应限流或排队而非无界耗尽资源。

### NFR-029 用户资源配额

- **优先级：** Must
- **版本：** MVP
- **要求：** 单个注册用户默认最多保存 100 个活动命盘；额度可配置，归档命盘和企业额度在后续版本单独定义。
- **验收标准：** 达到上限时禁止继续创建并提供归档或删除引导；配额检查具备并发一致性；管理员调整额度有审计且不删除既有命盘。

### NFR-030 内容长度边界

- **优先级：** Must
- **版本：** MVP
- **要求：** MVP 单份在线报告正文上限为 30,000 个 Unicode 字符，单条 AI 对话回答上限为 8,000 个 Unicode 字符；具体主题应使用结构化分页或折叠，而非无限增长。
- **验收标准：** 超限内容在生成阶段被安全压缩、拆分或拒绝，不得截断证据标识和风险提示；打印测试覆盖最大允许报告；V1 PDF 可另行定义页数上限。

## 25C. 独立领域状态机与端到端进度需求

### Chart 状态

`Draft → Validating → Validated → Calculating → Calculated → Archived`

- Chart 只负责确定性输入验证、排盘计算及 CalculationSnapshot 生命周期，不拥有规则分析、证据构建、AI 解读或报告生成状态。
- `Validating` 失败进入 `ValidationFailed`；输入修正并重新确认后，可按合法流程重新进入验证。
- `Calculating` 失败进入 `CalculationFailed`；关键交叉验证差异进入 `VerificationBlocked`。问题解决后，可使用锁定口径重新发起计算。
- 只有存在有效 CalculationSnapshot 时，Chart 才可进入 `Calculated` 并发布可供下游使用的确定性事实。
- `Calculated` 是 Chart 的计算完成状态，不表示规则、证据、AI、报告或端到端分析已经完成。
- RuleRun、EvidenceBundle、AIAnalysis、Timeline 或 Report 的成功、失败、降级或归档不得修改 Chart 状态。
- `Archived` 不等于删除；恢复归档不得改变原 CalculationSnapshot 或版本历史。

### RuleRun 状态

`Requested → Running → Validating → Completed / Failed`

- 正式 RuleRun 只能基于有效 CalculationSnapshot 和 Published RuleSetVersion 创建。
- RuleRun 管理本次执行的 RuleFinding、冲突关系和完整性；规则执行状态不得写入 Chart。
- `Completed` 只表示本次规则执行已通过完整性验证，不表示证据、AI 或报告完成。
- `Failed` 只影响本次 RuleRun；不得把关联 Chart 从 `Calculated` 改为失败状态。

### EvidenceBundle 状态

`Building → Validating → Frozen / Invalid`

- EvidenceBundle 管理同一分析范围内的 Evidence，并锁定 CalculationSnapshot、RuleRun、KnowledgeCitation 和证据策略版本。
- 只有通过引用、范围、版本和支持关系验证的 Bundle 才可进入 `Frozen`。
- `Frozen` 后不得新增或修改 Evidence；需要修订时必须创建新的 EvidenceBundle。
- `Invalid` 不得触发正式 AIAnalysis 或 Frozen Report，也不得修改 Chart 或 RuleRun 状态。

### AIAnalysis 状态

`Planned → Generating → Validating → Completed / Rejected / Failed`

- AIAnalysis 只能基于允许范围内的确定性事实和 Frozen EvidenceBundle 生成结构化解释。
- AIAnalysis 不计算命盘事实、不创建 Evidence、不扩大 AnalysisPlan，也不修改 Chart、RuleRun 或 EvidenceBundle。
- 模型供应商原始输出不是正式领域结果；只有通过结构、事实、引用、冲突和风险校验后才可进入 `Completed`。
- `Rejected` 或 `Failed` 只表示本次 AIAnalysis 未形成正式结果。系统仍可展示已验证的确定性排盘、规则和证据，并明确 AI 能力不可用或已降级。

### Report 状态

`Queued → Generating → Validating → Completed → Frozen → Archived`

- `Generating` 或 `Validating` 可进入 `Failed`。
- 报告只有通过结构、事实、引用和风险检查后才可进入 `Completed`。
- 正式展示前立即冻结；`Frozen` 内容不可原地修改。
- 重新分析创建新报告，新报告可将旧报告标记为 `Superseded`，但不得删除或覆盖旧内容。
- V1 对外分享是冻结报告上的独立发布状态，不改变报告正文状态。
- Report 的 `Completed` 或 `Frozen` 只属于 Report Aggregate，不得作为 Chart 状态或反向修改 Chart。

### AnalysisProgress 只读 Projection

- `AnalysisProgress` 汇总 Chart、RuleRun、EvidenceBundle、AIAnalysis、Timeline 和 Report 的公开状态，用于展示 End-to-End Analysis Flow 的当前进度。
- `AnalysisProgress` 是只读、可重建、最终一致的查询 Projection，不是 Entity、Aggregate Root、Process Manager 或 Saga。
- `AnalysisProgress` 不拥有任何参与对象的领域状态，不发布用于推进业务流程的领域决定，也不得反向修改任何 Aggregate。
- Projection 延迟、缺失或重建不得被解释为源 Aggregate 状态改变；用户界面必须能够表示处理中、失败、降级和投影延迟。
- 整体处理完成应通过 Frozen Report、明确的流程终态或 AnalysisProgress 对独立状态的汇总表达，不得重新引入 Chart 的跨 Context 完成状态。

### End-to-End Analysis Flow

1. BirthInput 确认后，Chart 完成验证和确定性计算并保持 `Calculated`。
2. 有效 CalculationSnapshot 可触发独立 RuleRun；规则执行不会改变 Chart 状态。
3. Completed RuleRun 及允许知识可用于构建并冻结 EvidenceBundle。
4. Frozen EvidenceBundle 可触发受范围、授权和风险策略约束的 AIAnalysis；AI 不可用时保留确定性能力。
5. Report 按报告类型读取兼容的 Snapshot、Frozen EvidenceBundle、所需 Completed AIAnalysis 和 Timeline，并在通过校验后冻结。
6. 跨 Context 的步骤由已批准的应用流程、事件和 Process Manager / Saga 推进；AnalysisProgress 只读取并汇总结果。
7. 任一下游步骤失败、拒绝或降级，都不得把 Chart 从 `Calculated` 改为其他下游状态。

### 规则、知识与 Prompt 状态

`Draft → InReview → Approved → Published → Deprecated → Retired`

- `Rejected` 可从 `InReview` 产生并返回新草稿修订。
- 只有 `Published` 版本可用于新正式分析。
- `Deprecated` 可为兼容期继续提供受控使用；`Retired` 不用于新任务。
- 状态转换必须符合 FR-036、FR-042、FR-058 和 FR-101 的职责分离要求。

## 25D. 统一错误分类规范

所有用户可见或 API 可见错误必须包含稳定错误码、可本地化安全消息、关联 ID、是否可重试和适当的 HTTP/传输状态；不得向用户暴露堆栈、SQL、模型原始响应、密钥或敏感内部信息。

| 前缀 | 范围 | 示例语义 |
|---|---|---|
| AUTH-xxx | 登录、会话、权限、资源归属 | 未认证、会话过期、无权访问 |
| CONSENT-xxx | 年龄、同意和授权 | 必要同意缺失、授权已撤回 |
| INPUT-xxx | 用户输入与地点解析 | 字段无效、地点歧义、时间精度不足 |
| TIME-xxx | 时区、夏令时、真太阳时和边界 | 本地时间不存在、偏移重复、边界待确认 |
| CALC-xxx | 排盘与算法 | 计算失败、版本不可用、交叉验证不一致 |
| RULE-xxx | 规则执行与冲突 | 无适用规则、执行错误、规则版本停用 |
| EVIDENCE-xxx | 证据与引用 | 引用不存在、证据不支持、权限不符 |
| KNOWLEDGE-xxx | 知识检索与授权 | 来源停用、授权过期、无匹配内容 |
| AI-xxx | 模型、结构、事实和安全检查 | 超时、结构无效、事实不一致、风险拒绝 |
| REPORT-xxx | 报告和打印 | 生成失败、未冻结、版本不可复现 |
| PRIVACY-xxx | 导出、删除、保留 | 核验失败、处理中、部分失败 |
| RATE-xxx | 配额、限流和容量 | 命盘上限、对话额度、系统繁忙 |
| SYSTEM-xxx | 未归类内部或依赖故障 | 临时不可用、任务失败 |

错误码只能新增或弃用，不得在同一主版本中改变既有码的语义。详细码表由架构文档规定责任模块，由后续接口规范列出具体编号。

## 25E. 测试策略

| 测试类型 | 最低范围 | 发布作用 |
|---|---|---|
| 单元测试 | 时间、历法、四柱、派生事实、规则条件、权限、状态转换、证据验证 | 阻断回归缺陷进入集成环境 |
| 契约测试 | 模块接口、AI 供应商适配器、地点服务、对象存储及未来 API | 防止供应商或模块变更破坏边界 |
| 黄金命例测试 | 专家批准的标准命例与预期事实 | 排盘和正式规则发布门禁 |
| 边界测试 | 节气、换日、闰月、时区、夏令时、真太阳时和不确定时间 | 排盘发布门禁 |
| 规则回归测试 | 每个规则版本的适用、不适用、冲突和信息不足样例 | 规则发布门禁 |
| AI 回归评测 | 事实一致、引用有效、冲突表达、拒答、高风险和多语言样例 | Prompt、模型和知识版本发布门禁 |
| 安全测试 | 身份、会话、越权、注入、敏感信息、后台职责分离和供应链 | 上线与重大变更门禁 |
| 隐私测试 | 最小化、去标识化、导出、删除、撤回、保留和日志扫描 | 上线与数据用途变更门禁 |
| 性能与压力测试 | NFR-025 至 NFR-030、排队、限流、恢复和资源耗尽 | 容量上线门禁 |
| 恢复演练 | 数据库、对象、任务、供应商故障和备份恢复 | 上线及定期运营门禁 |
| 可用性测试 | 三分钟旅程、术语理解、边界告知、移动端与辅助技术 | 产品上线门禁 |
| 人工抽样 | AI 解释质量、文化语境、风险措辞和版权引用 | 自动评测的必要补充 |

每个生产版本应保存测试集版本、执行环境、结果、例外审批和责任人。命理黄金命例的预期值只能由获授权专家确认；AI 测试通过不代表命理预测有效。

## 26. 已确认需求

以下需求已由上游产品愿景及本轮指令确认为基线：

1. 产品面向中国大陆简体中文成年用户，定位为传统文化分析与自我反思工具。
2. 普通用户易用性优先于功能数量，匿名试算不以注册为前提。
3. 排盘算法、规则结论和 AI 解释必须分离。
4. MVP 采用一套正式参考排盘算法，并进行内部独立交叉验证。
5. MVP 使用经专家确认的子平基础框架，不擅自确定争议规则。
6. 不确定出生时间必须显式表达。
7. 真太阳时默认关闭并可切换；子时换日可配置。
8. AI 对话属于 MVP，但限于当前命盘、未来三年和已支持主题。
9. 重要 AI 结论必须受事实和证据约束。
10. 证据等级不代表未来事件概率。
11. MVP 提供在线报告和打印；PDF 与分享链接进入 V1。
12. MVP 提供当前大运和未来三年；完整人生时间轴进入 V1。
13. 数据最小化、第三方 AI 去标识化和单独命例授权属于 MVP 必须要求。
14. 规则、知识、Prompt、模型和报告均需版本化、审计和可回滚或可复现。
15. 第一阶段采用模块化单体；插件只预留内部契约，不执行第三方动态代码。
16. MVP 不处理真实支付。
